# 为终端构建高效的 AI 编程代理：脚手架、治理框架、上下文工程及经验教训

## 摘要

AI 编程辅助领域正在经历从复杂 IDE 插件向通用终端原生代理的根本性转变。CLI 代理直接运行在开发者管理源代码、执行构建和部署环境的位置，为长期开发任务提供了前所未有的自主性。在本文中，我们提出了 OpenDev，一个用 Rust 编写的开源命令行编程代理，专为这一新范式而设计。有效的自主辅助需要严格的安全控制和高效的上下文管理，以防止上下文膨胀和推理退化。OpenDev 通过复合 AI 系统架构、工作负载专用模型路由、双代理架构（将规划与执行分离）、延迟工具发现，以及随时间推移逐步压缩旧观察结果的自适应上下文压缩来克服这些挑战。此外，它还采用自动记忆系统来跨会话累积项目特定知识，并通过事件驱动的系统提醒来对抗指令衰减。通过强制执行明确的推理阶段并优先考虑上下文效率，OpenDev 为终端优先 AI 辅助提供了一个安全、可扩展的基础，为健壮的自主软件工程提供了蓝图。

## 2 系统架构

### 2.1 概述

OpenDev 的架构涵盖四个主要层：入口与 UI 层、Agent 层、工具与上下文层、以及持久化层。用户查询按照顺序流经这一管道，从入口点经过 Agent 推理和工具执行，然后结果被持久化并呈现。

#### 入口与 UI 层

CLI 入口点解析参数并引导四个共享管理器（ConfigManager、SessionManager、ModeManager 和 ApprovalManager），这些管理器被注入到所有下游组件中。OpenDev 支持两个前端：一个是基于 Textual 构建的 TUI，使用阻塞式模态审批；另一个是由 FastAPI 和 WebSockets 支持的 Web UI，使用异步轮询审批。两者均实现了共享的 UICallback 协议，使 Agent 层与 UI 无关。

#### Agent 层

OpenDev 为不同的 LLM 分配五个专门的模型角色（第 2.2.5 节），每个角色都是延迟初始化的，并受本地缓存的能力注册表所驱动。该系统以两种模式运行：具有完整读写工具访问权限的普通模式（Normal Mode），以及仅限于只读工具的安全规划模式（Plan Mode）（第 2.2 节）。推理通过扩展 ReAct 循环（第 2.2.6 节）进行，每轮运行四个阶段：当 token 预算接近耗尽时的自动上下文压缩；可选的思考阶段，用于在可配置深度下进行行动前推理；可选的自我批评阶段；以及标准的 Reason-Act-Execute-Observe 行动阶段。

#### 工具与上下文层

工具执行层围绕 ToolRegistry 构建，将调用分派到涵盖文件操作、进程执行和网络访问的类型化处理器，支持批量并行执行和按需 MCP 工具发现（第 2.4.7 节）。技能（Skills）系统延迟注入可重用的、领域特定的三级层次结构提示模板（内置、项目、用户）。上下文工程层通过四个子系统管理 LLM 上下文窗口：系统提醒（第 2.3.4 节）用于上下文感知的行为指导，提示合成器用于模块化系统提示组装，记忆用于跨会话连续性，以及压缩（第 2.3.6 节）用于回收 token 预算。

#### 持久化层

OpenDev 通过四个存储库持久化状态：Config Manager 通过项目本地、用户全局、环境变量和内置默认值的层次结构解析设置；Session Manager 将完整对话历史保存为 JSON；Provider Cache 在本地存储模型能力元数据；操作日志跟踪文件更改以支持回滚。

#### 安全架构

由于 Agent 可以执行任意 shell 命令、覆盖文件并生成持久进程，单一安全机制是不够的。因此，OpenDev 采用纵深防御架构，包含五个独立的安全层，每一层都设计用于独立防止一类危害，从而使任何单一故障点都不会危及系统。
