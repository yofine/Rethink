重新定义 Harness：Agent = Model + Harness

TLDR

Harness 工程是我们围绕模型构建系统以将其转化为工作引擎的方式。模型包含智能，而 harness 使这种智能变得有用。我们定义了什么是 harness，并推导出当今和未来代理所需的核心组件。

为什么我们需要 Harness……从模型的视角

有些事情我们希望代理去做，但模型无法直接完成。这就是 harness 的用武之地。

模型（大多）接收文本、图像、音频、视频等数据，并输出文本，就这样。它们无法：

  • 在交互之间维持持久状态
  • 执行代码
  • 访问实时知识
  • 设置环境并安装包来完成工作

这些都是 harness 级别的特性。LLM 的结构需要某种 machinery 来包装它们以完成有用的工作。

例如，为了获得「聊天」的产品体验，我们将模型包装在一个 while 循环中，以跟踪之前的消息并追加新的用户消息。阅读本文的每个人都使用过这种 harness。主要想法是我们希望将期望的代理行为转化为 harness 中的实际功能。

用于持久存储和上下文管理的文件系统

我们希望代理拥有持久存储，以便与真实数据交互、卸载不适合放在上下文中的信息，并在会话之间持久化工作。

模型只能直接操作其上下文窗口内的知识。在文件系统出现之前，用户必须将内容直接复制/粘贴到模型中，这是一种笨拙的用户体验，对自主代理不起作用。世界已经在使用文件系统来工作，因此模型自然接受了数十亿个 token 的训练来学习如何使用它们。自然而然的解决方案变成了：

Harness 附带文件系统抽象和 fs-ops 工具。

文件系统可以说是最基础的 harness 原语，因为它解锁了以下能力：

  • 代理获得读取数据、代码和文档的工作空间。
  • 工作可以增量添加和卸载，而不是将所有内容都放在上下文中。代理可以存储中间输出并维护超越单个会话的状态。
  • 文件系统是自然的协作表面。多个代理和人类可以通过共享文件进行协调。像代理团队这样的架构就依赖于此。
  • Git 为文件系统添加了版本控制，因此代理可以跟踪工作、回滚错误和分支实验。

沙箱和用于执行和验证工作的工具

代理需要一个具有正确默认设置的环境，以便安全地行动、观察结果并取得进展。

我们已经赋予了模型存储和执行代码的能力，但所有这些都需要在某个地方发生。在本地运行代理生成的代码是有风险的，而且单个本地环境无法扩展到大规模代理工作负载。

沙箱为代理提供安全的操作环境。Harness 可以连接到沙箱来运行代码、检查文件、安装依赖并完成任务。这创建了安全的代码隔离执行。为了更高的安全性，harness 可以允许列出命令并强制执行网络隔离。沙箱还解锁了可扩展性，因为环境可以按需创建、分配到许多任务，并在工作完成后拆除。

好的环境配备好的默认工具。Harness 负责配置工具，以便代理可以完成有用的工作。这包括预装语言运行时和包、用于 git 和测试的 CLI、用于网络交互和验证的浏览器。

浏览器、日志、截图和测试运行器等工具为代理提供观察和分析工作的方式。这帮助它们创建自我验证循环，在其中编写应用程序代码、运行测试、检查日志并修复错误。

模型不会自行配置其执行环境。决定代理在哪里运行、有哪些可用工具、可以访问什么以及如何验证工作都是 harness 级别的设计决策。

对抗上下文腐败

代理的性能不应在工作过程中下降。

上下文腐败描述了模型在上下文窗口填满时推理和完成任务能力变差的情况。上下文是珍贵且稀缺的资源，因此 harness 需要策略来管理它。

今天的 harness 基本上是良好上下文工程的传递机制。

压缩解决了上下文窗口接近填满时该怎么办的问题。如果没有压缩，当对话超过上下文窗口时会发生什么？一种选择是 API 报错，这可不好。Harness 必须为此使用某种策略。因此，压缩智能地卸载和总结现有上下文窗口，以便代理可以继续工作。

工具调用卸载有助于减少大型工具输出的影响，这些输出可能会杂乱地填满上下文窗口而不提供有用的信息。Harness 保留超过 token 阈值数量的工具输出的头部和尾部 token，并将完整输出卸载到文件系统，以便模型在需要时可以访问它。

Skills 解决了在代理启动时将太多工具或 MCP 服务器加载到上下文中而导致性能下降的问题。Skills 是一种 harness 级别的原语，通过渐进式披露来解决这个问题。

Harness 的未来

模型训练和 Harness 设计的耦合

今天的代理产品如 Claude Code 和 Codex 在训练时将模型和 harness 纳入循环。这帮助模型改善 harness 设计者认为它们应该天然擅长的动作，如文件系统操作、bash 执行、使用子代理规划或并行化工作。

这创造了一个反馈循环。有用的原语被发现、添加到 harness 中，然后在训练下一代模型时使用。随着这个循环的重复，模型在它们训练的 harness 中变得更强大。

但这种共同进化对泛化有有趣的副作用。它以改变工具逻辑导致模型性能下降的方式出现。一个很好的例子在 Codex-5.3 提示指南中描述了 apply_patch 工具逻辑来编辑文件。一个真正智能的模型在 patch 方法之间切换应该没有问题，但在 harness 循环中训练会产生这种过拟合。

但这并不意味着最适合你的任务的 harness 是模型后训练时使用的 harness。Terminal Bench 2.0 排行榜就是一个很好的例子。Opus 4.6 在 Claude Code 中的得分远低于 Opus 4.6 在其他 harness 中。在之前的博客中，我们展示了如何仅通过更改 harness 将我们的编码代理从 Terminal Bench 2.0 的 Top 30 提升到 Top 5。在为你的任务优化 harness 方面还有很多可以挖掘的空间。

Harness 工程走向何方

随着模型变得更强大，今天存在于 harness 中的一些东西将被吸收到模型中。例如，模型将在规划、自我验证和长期连贯性方面变得更好，从而减少对上下文注入的需求。

这表明 harness 应该随着时间推移变得不那么重要。但正如提示工程在今天仍然有价值一样，harness 工程在构建好的代理方面很可能将继续有用。

诚然，今天的 harness 弥补了模型的缺陷，但它们也围绕模型智能工程系统以使其更有效。配置良好的环境、正确的工具、持久的状态和验证循环使任何模型都更高效，无论其基础智能如何。

Harness 工程是一个非常活跃的研究领域，我们用它来改进我们在 LangChain 的 harness 构建库 deepagents。以下是我们目前正在探索的一些开放且有趣的问题：

  • 编排数百个代理在共享代码库上并行工作
  • 分析自己的痕迹以识别和修复 harness 级别的故障模式的代理
  • 动态组装正确工具和上下文的 harness，正好及时完成给定任务，而不是预先配置
