# 重新定义 Harness：Agent = Model + Harness

## TLDR

Harness 工程是我们围绕模型构建系统以将其转化为工作引擎的方式。模型包含智能，而 harness 使这种智能变得有用。我们定义了什么是 harness，并推导出当今和未来代理所需的核心组件。

---

## 能否请人定义一下「Harness」？

### Agent = Model + Harness

**如果你不是模型，你就是 harness。**

Harness 是除了模型本身之外的每一段代码、配置和执行逻辑。原始模型不是代理。但当 harness 赋予它状态、工具执行、反馈循环和可强制执行的约束时，它就变成了代理。

具体来说，harness 包括以下内容：

- 系统提示
- 工具、Skills、MCP 及其描述
- 捆绑的基础设施（文件系统、沙箱、浏览器）
- 编排逻辑（子代理生成、切换、模型路由）
- 用于确定性执行的 Hooks/中间件（压缩、继续、lint 检查）

将代理系统的边界在模型和 harness 之间划分有很多种方式。但在我看来，这是最清晰的定义，因为它迫使我们围绕模型智能设计系统。

本文其余部分将介绍核心 harness 组件，并从模型这一核心原语出发，推导每个组件存在的原因。

---

## 为什么我们需要 Harness……从模型的视角

有些事情我们希望代理去做，但模型无法直接完成。这就是 harness 的用武之地。

模型（大多）接收文本、图像、音频、视频等数据，并输出文本，就这样。它们无法：

- 在交互之间维持持久状态
- 执行代码
- 访问实时知识
- 设置环境并安装包来完成工作

这些都是 harness 级别的特性。LLM 的结构需要某种 machinery 来包装它们以完成有用的工作。

例如，为了获得「聊天」的产品体验，我们将模型包装在一个 while 循环中，以跟踪之前的消息并追加新的用户消息。阅读本文的每个人都使用过这种 harness。主要想法是我们希望将期望的代理行为转化为 harness 中的实际功能。

---

## 从期望的代理行为反向推导 Harness 工程

Harness 工程帮助人类注入有用的先验知识来指导代理行为。随着模型变得更强大，harness 被用于外科手术式地扩展和纠正模型，以完成以前不可能完成的任务。

我们不会详尽列出每一个 harness 功能。目标是从帮助模型完成有用工作的起点推导出一组功能。我们将遵循这样的模式：**我们想要（或想要修复）的行为 → 帮助模型实现这一目标的 Harness 设计。**

---

### 用于持久存储和上下文管理的文件系统

我们希望代理拥有持久存储，以便与真实数据交互、卸载不适合放在上下文中的信息，并在会话之间持久化工作。

模型只能直接操作其上下文窗口内的知识。在文件系统出现之前，用户必须将内容直接复制/粘贴到模型中，这是一种笨拙的用户体验，对自主代理不起作用。世界已经在使用文件系统来工作，因此模型自然接受了数十亿个 token 的训练来学习如何使用它们。自然而然的解决方案变成了：

> **Harness 附带文件系统抽象和 fs-ops 工具。**

文件系统可以说是最基础的 harness 原语，因为它解锁了以下能力：

- 代理获得读取数据、代码和文档的工作空间。
- 工作可以增量添加和卸载，而不是将所有内容都放在上下文中。代理可以存储中间输出并维护超越单个会话的状态。
- 文件系统是自然的协作表面。多个代理和人类可以通过共享文件进行协调。像代理团队这样的架构就依赖于此。
- Git 为文件系统添加了版本控制，因此代理可以跟踪工作、回滚错误和分支实验。

---

### Bash + 代码作为通用工具

我们希望代理能够自主解决问题，而不需要人类预先设计每个工具。

今天主要的代理执行模式是 ReAct 循环，在这种循环中，模型通过工具调用进行推理、执行动作、观察结果，然后在 while 循环中重复。但 harness 只能执行它们有逻辑的工具。与其强迫用户为每个可能的动作构建工具——更好的解决方案是给代理一个像 bash 这样的通用工具。

> **Harness 附带 bash 工具，这样模型可以通过编写和执行代码自主解决问题。**

Bash + 代码执行是让模型拥有一台电脑并让它自主弄清楚其余事情的重要一步。模型可以通过代码即时设计自己的工具，而不是被限制在预先配置好的固定工具集中。

---

### 沙箱和用于执行和验证工作的工具

代理需要一个具有正确默认设置的环境，以便安全地行动、观察结果并取得进展。

我们已经赋予了模型存储和执行代码的能力，但所有这些都需要在某个地方发生。在本地运行代理生成的代码是有风险的，而且单个本地环境无法扩展到大规模代理工作负载。

**沙箱**为代理提供安全的操作环境。Harness 可以连接到沙箱来运行代码、检查文件、安装依赖并完成任务。这创建了安全的代码隔离执行。为了更高的安全性，harness 可以允许列出命令并强制执行网络隔离。沙箱还解锁了可扩展性，因为环境可以按需创建、分配到许多任务，并在工作完成后拆除。

好的环境配备好的默认工具。Harness 负责配置工具，以便代理可以完成有用的工作。这包括预装语言运行时和包、用于 git 和测试的 CLI、用于网络交互和验证的浏览器。

浏览器、日志、截图和测试运行器等工具为代理提供观察和分析工作的方式。这帮助它们创建自我验证循环，在其中编写应用程序代码、运行测试、检查日志并修复错误。

模型不会自行配置其执行环境。决定代理在哪里运行、有哪些可用工具、可以访问什么以及如何验证工作都是 harness 级别的设计决策。

---

### 用于持续学习的记忆和搜索

代理应该记住它们见过什么，并访问训练时不存在的信息。

模型除了权重和当前上下文中的内容外，没有额外的知识。在无法访问编辑模型权重的情况下，「添加知识」的唯一方式是通过上下文注入。

对于记忆，文件系统再次成为核心原语。Harness 支持像 AGENTS.md 这样的记忆文件标准，在代理启动时注入到上下文中。当代理添加和编辑此文件时，harness 将更新后的文件加载到上下文中。这是一种持续学习形式，代理在一个会话中持久存储知识，并将其注入到未来会话中。

知识截止意味着模型无法直接访问新数据，比如更新后的库版本，除非用户直接提供它们。对于最新知识，Web Search 和像 Context7 这样的 MCP 工具帮助代理访问超出知识截止日期的信息，如新库版本或训练停止时不存在的当前数据。

Web Search 和用于查询最新上下文的工具是值得内置到 harness 中的有用原语。

---

### 对抗上下文腐败

代理的性能不应在工作过程中下降。

**上下文腐败**描述了模型在上下文窗口填满时推理和完成任务能力变差的情况。上下文是珍贵且稀缺的资源，因此 harness 需要策略来管理它。

今天的 harness 基本上是良好上下文工程的传递机制。

**压缩**解决了上下文窗口接近填满时该怎么办的问题。如果没有压缩，当对话超过上下文窗口时会发生什么？一种选择是 API 报错，这可不好。Harness 必须为此使用某种策略。因此，压缩智能地卸载和总结现有上下文窗口，以便代理可以继续工作。

**工具调用卸载**有助于减少大型工具输出的影响，这些输出可能会杂乱地填满上下文窗口而不提供有用的信息。Harness 保留超过 token 阈值数量的工具输出的头部和尾部 token，并将完整输出卸载到文件系统，以便模型在需要时可以访问它。

**Skills** 解决了在代理启动时将太多工具或 MCP 服务器加载到上下文中而导致性能下降的问题。Skills 是一种 harness 级别的原语，通过渐进式披露来解决这个问题。

---

### 长时间自主执行

我们希望代理能够自主、正确地完成复杂的长期工作。

自主软件创作是编码代理的圣杯。但今天的模型存在早期停止、分解复杂问题困难以及工作跨越多个上下文窗口时的不连贯性问题。一个好的 harness 必须围绕所有这些进行设计。

这正是早期 harness 原语开始复合的地方。长期工作需要持久状态、规划、观察和验证，以在多个上下文窗口中保持工作。

- **用于跨会话跟踪工作的文件系统和 Git。** 代理在长期任务中产生数百万个 token，因此文件系统能够持久地捕获工作以跟踪进度。添加 Git 允许新代理快速了解项目的最新工作和历史。对于多个代理协同工作，文件系统还充当工作共享账本，代理可以以此协作。

- **用于继续工作的 Ralph Loop。** Ralph Loop 是一种 harness 模式，通过 hook 拦截模型的退出尝试，并在干净的上下文窗口中重新注入原始提示，迫使代理继续针对完成目标工作。文件系统使这成为可能，因为每次迭代从干净的上下文开始，但从上一次迭代读取状态。

- **规划和自我验证以保持正轨。** 规划是模型将目标分解为一系列步骤的时候。Harness 通过良好的提示和注入如何使用文件系统中的计划文件的提醒来支持这一点。在完成每个步骤后，代理可以通过自我验证检查其工作的正确性。Harness 中的 hook 可以运行预定义的测试套件，并在失败时将错误信息循环回模型，或者模型可以被提示独立评估其代码。验证将解决方案根植于测试中，并为自我改进创建反馈信号。

## Harness 的未来

### 模型训练和 Harness 设计的耦合

今天的代理产品如 Claude Code 和 Codex 在训练时将模型和 harness 纳入循环。这帮助模型改善 harness 设计者认为它们应该天然擅长的动作，如文件系统操作、bash 执行、使用子代理规划或并行化工作。

这创造了一个反馈循环。有用的原语被发现、添加到 harness 中，然后在训练下一代模型时使用。随着这个循环的重复，模型在它们训练的 harness 中变得更强大。

但这种共同进化对泛化有有趣的副作用。它以改变工具逻辑导致模型性能下降的方式出现。一个很好的例子在 Codex-5.3 提示指南中描述了 apply_patch 工具逻辑来编辑文件。一个真正智能的模型在 patch 方法之间切换应该没有问题，但在 harness 循环中训练会产生这种过拟合。

但这并不意味着最适合你的任务的 harness 是模型后训练时使用的 harness。Terminal Bench 2.0 排行榜就是一个很好的例子。Opus 4.6 在 Claude Code 中的得分远低于 Opus 4.6 在其他 harness 中。在之前的博客中，我们展示了如何仅通过更改 harness 将我们的编码代理从 Terminal Bench 2.0 的 Top 30 提升到 Top 5。在为你的任务优化 harness 方面还有很多可以挖掘的空间。

### Harness 工程走向何方

随着模型变得更强大，今天存在于 harness 中的一些东西将被吸收到模型中。例如，模型将在规划、自我验证和长期连贯性方面变得更好，从而减少对上下文注入的需求。

这表明 harness 应该随着时间推移变得不那么重要。但正如提示工程在今天仍然有价值一样，harness 工程在构建好的代理方面很可能将继续有用。

诚然，今天的 harness 弥补了模型的缺陷，但它们也围绕模型智能工程系统以使其更有效。配置良好的环境、正确的工具、持久的状态和验证循环使任何模型都更高效，无论其基础智能如何。

Harness 工程是一个非常活跃的研究领域，我们用它来改进我们在 LangChain 的 harness 构建库 deepagents。以下是我们目前正在探索的一些开放且有趣的问题：

- 编排数百个代理在共享代码库上并行工作
- 分析自己的痕迹以识别和修复 harness 级别的故障模式的代理
- 动态组装正确工具和上下文的 harness，正好及时完成给定任务，而不是预先配置
