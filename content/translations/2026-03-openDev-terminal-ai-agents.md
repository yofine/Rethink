# 为终端构建高效的 AI 编程代理：脚手架、治理框架、上下文工程及经验教训

## 摘要

AI 编程辅助领域正在经历从复杂 IDE 插件向通用终端原生代理的根本性转变。CLI 代理直接运行在开发者管理源代码、执行构建和部署环境的位置，为长期开发任务提供了前所未有的自主性。在本文中，我们提出了 OpenDev，一个用 Rust 编写的开源命令行编程代理，专为这一新范式而设计。有效的自主辅助需要严格的安全控制和高效的上下文管理，以防止上下文膨胀和推理退化。OpenDev 通过复合 AI 系统架构、工作负载专用模型路由、双代理架构（将规划与执行分离）、延迟工具发现，以及随时间推移逐步压缩旧观察结果的自适应上下文压缩来克服这些挑战。此外，它还采用自动记忆系统来跨会话累积项目特定知识，并通过事件驱动的系统提醒来对抗指令衰减。通过强制执行明确的推理阶段并优先考虑上下文效率，OpenDev 为终端优先 AI 辅助提供了一个安全、可扩展的基础，为健壮的自主软件工程提供了蓝图。

## 1 引言

大型语言模型（LLMs）的快速发展催生了软件开发的新范式：作为自主智能体运行的AI驱动编码助手[95, 105, 99]。与传统代码补全工具仅提供内联代码片段不同，代理式编码助手能够对复杂任务进行推理、执行多步骤计划，并通过工具使用与开发环境进行交互。综合调查记录了代码智能研究的爆发式增长[47, 46]，代理软件工程路线图[35]正式确定了人机协作的方法论原则。SWE-Agent [95]、OpenHands [84]和HyperAgent [69]等系统已在标准基准测试中展示了自主智能体的潜力。商业影响同样显著，GitHub Copilot吸引了超过1500万开发者[30]，AI原生编辑器的收入快速增长[4]，主要实验室纷纷推出自主编码智能体[18, 65]，这标志着代理式编码已从研究原型走向工业部署。

### 终端原生智能体的兴起

过去几年间，AI编码助手一直与IDE紧密集成，作为需要持续人工监督的被动协作助手。最近，一个重大转变开始显现——从复杂的IDE插件转向更简单的命令行界面。Claude Code [2]引领了这一转变，证明了终端原生智能体可以在实际软件工程任务中达到或超过IDE集成工具的水平。终端是软件开发的操作核心，原生支持源代码管理、构建系统、远程SSH会话和无头服务器环境。Aider [28]、CodeAct [83]和Open Interpreter [63]等早期系统展示了基于终端的AI结对编程和可执行代码操作的可行性。如今，每个主要AI实验室都提供了CLI智能体[2, 32, 65]，同时还有Goose [10]、OpenCode [19]和Crush [13]等开源替代方案。然而，实现这一潜力并非易事。Terminal-Bench [57]和LongCLI-Bench [25]等基准测试表明，即使是前沿模型在持续终端操作方面也存在困难，这凸显了针对特定工程解决方案的需求。

这些基准测试结果指向了任何长期运行的终端智能体必须解决的三个基本工程挑战：管理超过模型令牌预算的会话的有限上下文窗口、在智能体可以执行任意shell命令时防止破坏性操作，以及在不过度消耗智能体提示预算的情况下扩展能力。我们围绕两个阶段组织架构响应：脚手架（scaffolding）阶段，在第一个提示之前组装智能体（系统提示、工具模式、子智能体注册表）；以及线束（harness）阶段，在运行时协调工具分发、上下文管理和安全执行[102]（第2.2节）。然而，终端原生代理工具的设计空间在很大程度上仍未被探索：大多数生产系统是闭源的，没有记录架构决策，现有的开源框架要么针对基准测试而不是交互式使用，要么缺乏已发布的技术报告[56, 39]。三个关键开放问题推动这项工作：多模型架构如何在不同认知任务中平衡成本、延迟和能力？什么安全机制可以防止破坏性操作而不影响开发者生产力？系统如何在有限的上下文限制下维持长时间对话？

在本文中，我们介绍OpenDev，一个用于软件工程的开源AI驱动命令行代理。据我们所知，这是第一个关于开源、终端原生、交互式编码代理的综合技术报告。现有系统分为两类：面向基准测试的框架（如SWE-Agent [95]）已发表研究论文，但主要用于自动评估而非日常交互使用；OpenHands [84]是生产级且文档完善的，但通过基于浏览器的UI而非终端界面运行；CLI原生代理如Aider [28]、Goose [10]、OpenCode [19]、Crush [13]和Gemini CLI [32]缺乏记录其设计决策的已发布技术报告；Claude Code [2]是CLI原生的，但既非开源，也未附带已发布的技术报告。本文的目的是不是展示新颖的算法突破，而是分享设计决策、权衡以及从工程化一个生产就绪的代理编码系统中汲取的经验教训，从而弥合封闭源代码工业实践与开放学术讨论之间的差距。

OpenDev的一个核心设计原则是它是一个复合AI系统[103]：不是单一的LLM，而是结构化的代理和工作流集合，每个都独立绑定到用户配置的LLM（第2.2.5节）。Zaharia等人阐述的这一框架认为，最先进的AI结果越来越多地通过组合多个模型、检索器和工具的系统来实现，而不是依赖单一的模型调用。OpenDev将这一原则付诸实践：其解耦架构使系统天生与模型无关，学习到的模型路由等技术[62]可以在工作流层面应用。切换提供商或优化成本只需要配置更改，而不是代码更改。因此，系统功能在部署时不是固定的，而是可以随着更好的模型出现而持续升级。

### 设计原则与贡献

OpenDev的设计遵循三个总体原则。首先，关注点分离：每个架构决策（模型选择、上下文管理、安全执行、工具分发）应该可以独立配置和替换，而不影响其他决策。其次，渐进降级：系统应该在资源耗尽时优雅地运行，无论是令牌预算、迭代次数还是网络连接。第三，透明性优于魔法：每个系统操作（工具调用、安全否决、上下文压缩、内存更新）应该是可观察的，并且对开发者可覆盖。这些原则体现在五个具体贡献中：

- **1. 通过复合架构实现每工作流级别的 LLM 可配置性**。不同的执行阶段对模型能力、延迟和成本有着不同的需求。我们提出了一种每工作流独立的 LLM 绑定架构，其中每个认知工作流通过用户配置独立选择模型（第 2.2.5 节），这一设计基于复合 AI 系统视角 [103] 和模型路由研究 [62]。

- **2. 扩展的 ReAct 执行流水线**。我们扩展了标准的 ReAct 循环 [99]，引入了显式思考阶段和可选的自我批判阶段，将推理与行动分离（以下简称 ReAct 循环；第 2.2.6 节），并直接将分阶段上下文压缩集成到推理循环中，借鉴了上下文工程研究的洞察 [56, 39]。

- **3. 长时间跨度的行为引导**。我们提出了事件驱动的系统提醒机制，通过在决策点注入有针对性的引导来抵消长时间运行会话中的指令衰减，而非仅依赖于初始系统提示（第 2.3.4 节）。条件提示组合管道从独立的、按优先级排序的提示部分组装智能体指令，这些部分仅在上下文相关时加载，在减少提示开销的同时保留了全面的指导（第 2.3.1 节）。

- **4. Token 高效的可扩展性和深度防御安全**。我们提出了基于注册表的工具架构，通过 MCP [1] 实现延迟发现的外部工具（第 2.4.7 节），以及五层安全架构，在逐步降低的抽象层级上强制执行约束：提示级护栏、通过双智能体分离实现的模式级工具门控（第 2.2 节）、具有持久权限的运行时审批系统、工具级验证，以及用户定义的生命周期钩子（第 2.1 节）。

- **5. 将上下文工程作为一等公民**。我们将上下文管理视为一等工程问题 [3, 56]，提出了自适应上下文压缩（第 2.3.6 节）、用于对抗注意力衰减的事件驱动系统提醒（第 2.3.4 节），以及积累项目特定知识的经验驱动记忆管道 [66, 106]。我们的压缩和记忆策略与上下文工程最新理论工作中提出的熵减原则和最小充分性原则相一致 [39, 101]。

本文其余部分按照从构建到反思再到上下文的路径展开。第 2 节详细阐述了所构建的内容：跨越智能体推理、上下文工程、工具和持久层的四层系统架构。第 3 节探讨了所获得的经验：迭代开发过程中出现的五个跨领域设计张力及其可迁移的教训。第 4 节将这些决策置于更广泛的研究景观中，第 5 节指出了未来的方向。附录提供了工具、提示、配置模式和实现常量的参考级目录。

## 2 系统架构

### 2.1 概述

OpenDev 的架构涵盖四个主要层：入口与 UI 层、Agent 层、工具与上下文层、以及持久化层。用户查询按照顺序流经这一管道，从入口点经过 Agent 推理和工具执行，然后结果被持久化并呈现。

#### 入口与 UI 层

CLI 入口点解析参数并引导四个共享管理器（ConfigManager、SessionManager、ModeManager 和 ApprovalManager），这些管理器被注入到所有下游组件中。OpenDev 支持两个前端：一个是基于 Textual 构建的 TUI，使用阻塞式模态审批；另一个是由 FastAPI 和 WebSockets 支持的 Web UI，使用异步轮询审批。两者均实现了共享的 UICallback 协议，使 Agent 层与 UI 无关。

#### Agent 层

OpenDev 为不同的 LLM 分配五个专门的模型角色（第 2.2.5 节），每个角色都是延迟初始化的，并受本地缓存的能力注册表所驱动。该系统以两种模式运行：具有完整读写工具访问权限的普通模式（Normal Mode），以及仅限于只读工具的安全规划模式（Plan Mode）（第 2.2 节）。推理通过扩展 ReAct 循环（第 2.2.6 节）进行，每轮运行四个阶段：当 token 预算接近耗尽时的自动上下文压缩；可选的思考阶段，用于在可配置深度下进行行动前推理；可选的自我批评阶段；以及标准的 Reason-Act-Execute-Observe 行动阶段。

#### 工具与上下文层

工具执行层围绕 ToolRegistry 构建，将调用分派到涵盖文件操作、进程执行和网络访问的类型化处理器，支持批量并行执行和按需 MCP 工具发现（第 2.4.7 节）。技能（Skills）系统延迟注入可重用的、领域特定的三级层次结构提示模板（内置、项目、用户）。上下文工程层通过四个子系统管理 LLM 上下文窗口：系统提醒（第 2.3.4 节）用于上下文感知的行为指导，提示合成器用于模块化系统提示组装，记忆用于跨会话连续性，以及压缩（第 2.3.6 节）用于回收 token 预算。

#### 持久化层

OpenDev 通过四个存储库持久化状态：Config Manager 通过项目本地、用户全局、环境变量和内置默认值的层次结构解析设置；Session Manager 将完整对话历史保存为 JSON；Provider Cache 在本地存储模型能力元数据；操作日志跟踪文件更改以支持回滚。

#### 安全架构

由于 Agent 可以执行任意 shell 命令、覆盖文件并生成持久进程，单一安全机制是不够的。因此，OpenDev 采用纵深防御架构，包含五个独立的安全层，每一层都设计用于独立防止一类危害，从而使任何单一故障点都不会危及系统。
